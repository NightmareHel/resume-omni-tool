## What Was Discussed and Decided

### 1. Phase 1 implementation approach

Before writing any code, Sid approved a detailed plan (stored in the Claude plans system) covering four parts:

- **Part 1:** PDF generation using Playwright `page.setContent()` + `page.pdf()` (chosen over external PDF libraries since Playwright was already installed). ATS-safe single-column HTML template researched for maximum parse accuracy.
- **Part 2:** Apply worker PDF upload — generate PDF in `processApplication` before dispatch, pass path to all fillers, clean up in `finally`.
- **Part 3:** ATS-specific form fillers — separate `fillLever()` and `fillAshby()` functions using `aria-label*` and `_systemfield_*` selectors respectively (Greenhouse IDs don't exist on other ATS platforms). Custom question detection via counting visible text-like inputs vs known standard selectors; abort to `manual_required` if any unrecognized fields.
- **Part 4:** Cover letter generation — `generateCoverLetter()` added to `lib/claude.ts`, runs in parallel with gap analysis and resume rewrite in the tailor route. Plain text output, temp=0.4.

**Key decision:** Sid chose "Abort to manual" for custom screening questions rather than trying to answer them. Applications with custom questions get `status = 'manual_required'` and stop being retried. **This was a deliberate design choice** — do not change it without discussing with Sid.

**Key decision:** Apply worker stays manual-launch only. No PM2. Sid wants to supervise apply sessions.

### 2. Seniority filter

Sid said he was seeing too many senior roles in the scraped job board. We discovered the issue: the title filter in `_http.ts` only had an include list (`engineer`, `developer`, etc.) with no exclusion. A title like "Senior Software Engineer" matched "engineer" and got through.

Fix: Added `matchesTitleFilter()` in `_http.ts` that rejects any title matching 13 seniority/experience terms before checking the include list. Applied globally across all four scrapers.

After implementing, Sid also said the old senior jobs were still showing in the UI even though the scraper filter was now live. That's because the existing DB rows were unaffected. We ran a SQL archive pass (498 of 852 jobs archived) but Sid still saw old ones. He chose to do a **full wipe and fresh scrape** rather than a partial cleanup. Full wipe included: jobs, applications, email_threads, scrape_runs tables. 233 clean jobs came back from the re-scrape.

**During the re-scrape, three companies returned 404 errors:**

- Perplexity (`perplexity-ai` Ashby slug — may have changed ATS)

These three companies contributed zero jobs. This is a known outstanding issue. Someone needs to look up the correct slugs for each and update `config/scrapers.json`. No code changes needed — just slug string fixes.

### 3. My Jobs feature

Sid asked for a second section on the jobs page where he can add job links he finds independently (LinkedIn referrals, company career pages, jobs sent to him by people). He wanted:
- Auto-fetch title and company from the URL (not manual entry)
- One at a time (not batch paste)
- Full feature parity with scraped jobs (score, tailor, pipeline)

We went into plan mode, researched the existing `JobBoard` and `JobCard` components, and designed the implementation. Key architectural decision: manual jobs live in the same `jobs` table as scraped jobs, distinguished by `source = 'custom'`. The `custom` source was already listed in `JobFilters.tsx` (the old filter dropdown) so no schema changes were needed.

The import route uses Playwright with `waitUntil: 'networkidle'` to handle JS-rendered job pages (Greenhouse, Lever, Ashby pages are all client-side rendered and would return empty content with a plain fetch). Title/company extraction uses a layered heuristic: strip known ATS suffixes first, then try pipe-split, then dash-split, then fall back to `og:site_name` from meta. This handles most real-world job posting URLs.

## Architecture Notes for Next Session

### The `resume_text` format

The `resume_text` column in `applications` stores sections as double-newline-separated blocks. Each block starts with the section name on the first line, then content on subsequent lines. Example:

```
PROFESSIONAL SUMMARY
Experienced software engineer with a focus on AI systems...

EXPERIENCE
Software Engineer at Acme Corp | Jan 2024 - Present
- Built distributed data pipeline
- Reduced latency by 40%

SKILLS
Python, TypeScript, React, PostgreSQL
```

This format is produced by the tailor route: `rewrite.sections.map((s) => `${s.name.toUpperCase()}\n${s.rewritten}`).join('\n\n')`.

The PDF renderer in `lib/resume-pdf.ts` parses this format: split on `\n\n+`, first line of each chunk = section name, rest = content. Lines starting with `•` or `-` become `<li>` elements. Lines with ` | ` or ` – ` become bold entry headers.

### The `jobId` function

`jobId(source, externalId)` in `lib/ids.ts` is a SHA-256 hash truncated to 32 hex chars. For scraped jobs, externalId is the ATS-assigned job ID. For custom jobs (manual import), externalId is the full URL string. This is why the unique constraint on the `url` column also serves as dedup protection for manual imports — the `jobId` hash would also produce the same ID on duplicate URL submission.

### Why `excludeCustom` instead of filtering client-side

The `JobBoard` component already passes `source` as a filter to dedicate the board to a specific ATS. If we just filtered client-side, a user who had set `source=greenhouse` in filters would see custom jobs vanish, but a user with no source filter would see custom jobs mixed into the scraped board. The server-side `excludeCustom=true` ensures the scraped board never shows custom jobs regardless of what other filters are active.

### Playwright in Next.js API routes

Both `app/api/jobs/import/route.ts` and `app/api/applications/[id]/resume.pdf/route.ts` use Playwright directly inside Next.js API route handlers. This works because `playwright` and `playwright-core` are listed in `serverExternalPackages` in `next.config.ts`, which tells Next.js to treat them as Node.js-only packages and skip bundling. Without this, the `chromium.launch()` call would fail during the Next.js build or at runtime.

The timeout for Playwright in the import route is 30 seconds (`waitUntil: 'networkidle', timeout: 30_000`). Next.js API routes have no default timeout in dev but may time out in production environments. This is a known potential issue (item 12 in the broken/incomplete list in CONTEXT.md).

### State machine transition added: `pending → manual_required`

Before this session, `pending` could only go to `submitted` or `withdrawn`. The apply worker's abort path for custom questions had nowhere to go — it would have kept the application in `pending` and retried on every 2-minute poll. Adding `manual_required` as a terminal state solves this: once aborted, the application is dead to the worker and stays visible in the pipeline as "manual_required" for the user to handle manually.

The `TERMINAL_STATUSES` set in `lib/application-state.ts` is used by the pipeline UI to prevent further status transitions from terminal states. `manual_required` is in this set.
