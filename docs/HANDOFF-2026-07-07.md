## PDF Generation — Current State

Two PDF endpoints exist. Neither has been tested end-to-end since the detail page was built.

### Resume PDF (`GET /api/applications/[id]/resume.pdf`)
- Built in Session 3, used by the apply worker since then
- Template: `lib/resume-pdf.ts` `buildHTML()` — ATS-safe, single-column, Arial 10.5pt, 18pt name, 12pt section headers with border-bottom, bullet `<ul><li>`, flex layout
- Parses `resume_text` by splitting on `\n\n+` — first line of each chunk becomes section header, remaining lines become content
- Contact info pulled from `profile` row, not from resume_text

### Cover Letter PDF (`GET /api/applications/[id]/cover.pdf`)
- Built in Phase C (Session 6) — **completely untested**
- Template: `lib/resume-pdf.ts` `buildCoverHTML()` — Arial 11pt, 1in margins, name + contact header, paragraphs split on `\n\n+`
- Cover letter text comes from `applications.cover_letter` (set by tailor/retailor)

### How PDFs surface in the UI
- Detail page (`/pipeline/[id]`): embedded in an `<iframe>` in the left column
- Toggle between resume and cover with buttons above the iframe
- Cache-bust: `?v=${cacheKey}` on the iframe src, incremented on every save or re-tailor
- Download button: `<a href={pdfSrc} download>` — triggers browser download

## PDF Generation — Known Issues and Risks

### Issue 1: Cover letter paragraph splitting may fail
**Risk: High. Untested.**
The cover letter template splits on `/\n\n+/`. The AI (`generateCoverLetter`) streams plain text with paragraphs separated by `\n\n`. However:
- If the AI output uses single `\n` between paragraphs (which Groq/Llama sometimes does), the entire letter renders as one block
- No fallback: if the regex finds no double-newlines, the whole text is one `<p>` tag

**Fix needed:** Update `buildCoverHTML` to also split on single newlines that precede a capital letter, or normalize `\n` to `\n\n` as a pre-pass before splitting.

### Issue 2: No error handling in PDF route handlers
**Risk: Medium.**
Both `resume.pdf/route.ts` and `cover.pdf/route.ts` call `generateResumePDF` / `generateCoverLetterPDF` with no try/catch around the Playwright call. If Chromium is not installed or the launch fails:
- The route crashes with an unhandled exception
- Next.js returns a raw 500 with an HTML error page, not a JSON error
- The iframe shows a browser error page with no helpful message

**Fix needed:** Wrap Playwright calls in try/catch in both route handlers. Return a clean JSON 500 on failure.

### Issue 3: Iframe may show blank on slow PDF generation
**Risk: Medium.**
Playwright PDF generation takes 2-4 seconds on a cold browser launch. The iframe starts loading immediately when the detail page opens. During generation:
- The iframe shows a browser loading spinner (acceptable)
- If generation takes >30s (stuck Playwright), the iframe may show a timeout error
- No user-facing feedback that generation is in progress

**Fix needed:** Consider adding a visible loading overlay on the iframe while the PDF is generating. Could use a short `setTimeout` approach or an explicit loading state tied to `onLoad` on the iframe.

### Issue 4: Resume text format assumptions
**Risk: Medium.**
The `parseSections` function in `lib/resume-pdf.ts` assumes `resume_text` is structured as:
```
SECTION NAME
- bullet one
- bullet two

NEXT SECTION
content line
```
The AI rewrite produces this format (section name in all-caps on first line, bullets on subsequent lines). However:
- If the user manually edits `resume_text` in the editor and breaks the format, section parsing silently fails
- Empty sections (section header with no content lines) render as just a header with no body

**Fix needed (low priority):** Add a visual preview in the editor showing how sections will be parsed, or validate format on save.

### Issue 5: Markdown bleed-through in cover letter
**Risk: Medium. Untested.**
The AI is prompted to return "plain text only, no markdown, no headers." However, Groq/Llama-3.3 sometimes includes markdown formatting (`**bold**`, bullet `-`, headings `#`) especially if the JD or profile contains such formatting. The `esc()` function in the template HTML-escapes these characters, so they render as literal text rather than causing XSS — but they look wrong in the PDF (e.g., `**key skills**` visible as-is).

**Fix needed:** Add a markdown-strip pre-pass in `generateCoverLetterPDF` before rendering: strip `**`, `*`, `#`, and leading `-` from each line.

### Issue 6: Resume PDF — no page break handling
**Risk: Low (cosmetic).**
Long resumes (5+ sections with many bullets) may overflow the Letter page. The current template has no `page-break-inside: avoid` or `@page` rules to control where breaks occur. Content will just overflow to a second page naturally, which is fine for ATS systems, but section headers may end up at the bottom of a page with their content on the next.

**Fix needed (low priority):** Add `page-break-inside: avoid` to `section` elements in the resume CSS.

### Issue 7: `Content-Length` header may be wrong
**Risk: Low.**
Both PDF routes set `'Content-Length': String(pdfBuffer.length)`. `pdfBuffer` is a Node.js `Buffer`, so `.length` is byte count — correct. `new Uint8Array(pdfBuffer)` preserves the same byte count. This should be fine, but worth verifying the browser doesn't truncate the PDF on download.

## PDF Testing Protocol

Work through this in order before considering PDFs "done."

### Step 1: Verify Playwright binary
```bash
npx playwright install chromium
# Then verify:
npx tsx -e "const { chromium } = require('playwright'); chromium.launch({ headless: true }).then(b => { console.log('ok'); b.close(); })"
```

### Step 2: Tailor a job to get a draft application
- Go to /jobs, find a job with a description (source: greenhouse, ashby, or lever — Simplify has null descriptions)
- Click Score, then Tailor
- Wait for SSE completion, land on /pipeline?tailored=true

### Step 3: Open the detail page
- Click "Open" on the draft card
- Verify the resume PDF loads in the iframe within 5 seconds
- If blank: open DevTools Network tab, find the `/resume.pdf` request, check its response (200 with PDF bytes, or an error)

### Step 4: Test the cover letter PDF
- Click the "Cover Letter" toggle above the iframe
- Verify cover letter PDF loads
- Check: are paragraphs separated correctly, or is it one giant block? (Issue 1)
- Check: is there any visible `**` or `#` markdown in the output? (Issue 5)

### Step 5: Test save + refresh
- In the Resume tab: edit one word in the textarea
- Click "Save + Refresh PDF"
- Verify the iframe reloads (spinner appears briefly) and the edited word is visible in the new PDF

### Step 6: Test direct URL access
- Copy `/api/applications/[id]/resume.pdf` and open in new browser tab
- Verify it downloads/renders without error
- Do the same for `/api/applications/[id]/cover.pdf`

### Step 7: Test download button
- On the detail page, click "Download" for both resume and cover
- Verify both files download and open correctly in a PDF viewer

### Step 8: Error path
- Temporarily break profile: go to /profile, clear full_name, save
- Visit /api/applications/[id]/resume.pdf directly
- Should return JSON `{ "error": "Profile not found" }` or similar — not a raw crash (Issue 2)
- Restore profile
